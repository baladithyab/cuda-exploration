**Status.** Authored, compiles, correctness PASSES. No timed bench in this
cell (orchestrator runs `./bench` separately on idle GPU per cuda-exploration
session convention).

## What this is

A CUDA C++ port of `cuda-attn-gdn` (W1c, 417.7 GB/s) that replaces the
synchronous `LDG.E.128` state-row read with a software pipeline built on
**`cuda::pipeline<thread_scope_thread>` + `cuda::memcpy_async`**, lowered
to **`cp.async`** (SASS: `LDGSTS.E.BYPASS.128` + `LDGDEPBAR` + `DEPBAR.LE`)
on sm_120.

The Wave 22.8 investigation
([`docs/research/wave17-w1c-tma-vs-ldg128-investigation.md`](../docs/research/wave17-w1c-tma-vs-ldg128-investigation.md))
established that cuTile's 610 GB/s vs nvcc-W1c's 417.7 GB/s gap is **not**
TMA — neither kernel emits `UTMALDG`. cuTile uses Blackwell async-tx
barriers (`SYNCS.PHASECHK.TRANS64.TRYWAIT`) plus a 100 KiB warp-specialised
shared-memory pipeline. This cell tests whether the same gap can be closed
with the *portable* `cuda::pipeline` API on sm_120, which lowers to
`cp.async` — a different async-load mechanism but with the same
"decouple gmem-load from FFMA chain" effect.

## Design

| dimension | W1c (`cuda-attn-gdn`) | W22.9 (`cuda-attn-gdn-async`) |
|---|---|---|
| TPB | 16 (half-warp) | 16 (half-warp) — **same** |
| BLOCK_V | 64 | 64 — **same** |
| `S_in[k]` row read | direct `LDG.E.128` from gmem | `cuda::memcpy_async` → smem ring → smem read |
| ring buffer | n/a | 4 stages × VLANES × 16 B = 1 KiB |
| pipeline scope | n/a | `thread_scope_thread` |
| Pass 1 inner loop | load → multiply → FFMA | wait_prior → load_smem → multiply → FFMA → release → prefetch_k+N |
| Pass 2 (S_out write + o accum) | unchanged | unchanged (reads `smem_S` — same as W1c) |

The deliberate choice: **TPB and stripe layout are identical to W1c**, so
the only changed variable is the async pipeline. This isolates the
`cp.async` effect from the warp-specialisation effect (which is a separate
question for Wave 23+).

`alpha`-scale of `S_in` happens at **consume time**, not produce time,
because `cuda::memcpy_async` is a pure copy. Pass 2 reads from `smem_S`
in the same byte order as W1c, so the FFMA accumulation order is preserved
and we get **bit-identical** numerical residuals.

**Reading the table.**
- The 16 `LDG.E.128` state-row loads are gone, replaced 1:1 by 16
  `LDGSTS.E.BYPASS.128` instructions. This is the textbook lowering of
  `cuda::memcpy_async(dst_smem, src_gmem, aligned_size_t<16>(16), pipe)`
  on sm_120.
- The smem-side traffic increased (+8 `LDS.128`, +8 `STS.128`) as expected:
  cp.async data lives in smem briefly before consumption, costing one smem
  round-trip per row tile but freeing the LSU to run ahead of the FFMA chain.
- `STG.E.128 = 16` is preserved — Pass 2's `S_out` write path is unchanged.
  `cuda::memcpy_async` is gmem→smem only on this arch; the smem→gmem
  output stream stays synchronous, which is fine because writes are not
  on the critical path.
- `BSSY/BSYNC.RECONVERGENT = 4/4` (vs W1c's 2/2): two extra warp-divergent
  fast paths from the prologue/steady-state branch in the pipeline loop.
- **No `SYNCS.PHASECHK.TRANS64`, no `mbarrier`, no `FENCE.VIEW.ASYNC.S`**.
  This kernel uses cp.async dependency barriers (`LDGDEPBAR` /
  `DEPBAR.LE`), which is the **sm_80-class async path** (introduced with
  Ampere). cuTile uses the **sm_90-class async-tx-barrier path**, which is
  separately accessible via `cuda::barrier` / `cuda::aw_barrier` on
  sm_90+ but is **not** what `cuda::pipeline<thread_scope_thread>`
  selects. This is a known design point — see "Pitfalls" below.

## Expected GB/s — rough estimate

**Conservative:** 430-470 GB/s (0-13% over W1c's 417.7).
**Optimistic:** 480-540 GB/s (15-29% over W1c).
**Stretch (cuTile parity):** 560-610 GB/s (only if the LSU is the actual
bottleneck and 4 in-flight cp.async slots fully cover the ~400-cycle gmem
latency).

Why conservative is more likely:

1. **TPB=16 is half-warp.** Half the warp lanes are idle, costing ≥2× on
   instruction-issue throughput regardless of memory pipelining. cuTile
   runs at full-warp occupancy with warp-specialisation. W22.9 doesn't
   touch this dimension.
2. **L2 traffic is bounded by the same 64 KiB read per `bh` block.** The
   total bytes moved is identical to W1c; cp.async can only *hide* latency,
   not reduce traffic. If the kernel is bandwidth-bound and not
   latency-bound, the gain is minimal.
3. **cp.async dependency-barriers (`LDGDEPBAR`) are coarser than
   async-tx-barriers (`SYNCS.PHASECHK.TRANS64`).** They wait on a *count*
   of completed groups, not on byte-level transaction credits, so the
   producer-consumer overlap quantum is per-group (one row tile)
   rather than per-byte.

Why the win could land in the optimistic range:

- W1c's FFMA accumulator chain has a hard data dep on the result of
  `*(float4*)Sbh+...` — that's ~400 cycles unhidden. With 4 stages we have
  ~1600 cycles of in-flight load capacity, which dominates a single row's
  consume cost (~30-40 cycles for 4 FFMAs). The LSU should run well ahead.
- Blackwell's L2 has higher hit rates on cp.async than on synchronous
  loads in some published benchmarks (NVIDIA HPC Compiler docs, post-Ampere).

The Wave 22.8 investigation predicted that `cuda::pipeline` from CUDA C++
"would close the gap" — this cell is the empirical test. **Orchestrator
should run `./bench --warmup 5 --iters 100` and compare best GB/s to the
417.7 baseline.**

## Pitfalls (lessons for future async-pipeline work)

1. **`cuda::aligned_size_t<16>` matters.** Without the alignment hint,
   `cuda::memcpy_async(dst, src, sizeof(float4), pipe)` may lower to
   per-byte fallback (a memcpy loop) instead of `LDGSTS.E.BYPASS.128`.
   We asserted 16 B with `cuda::aligned_size_t<16>(sizeof(float4))` and
   verified the SASS shows the `.128` form.
2. **Thread-scope vs block-scope pipeline.** We used
   `cuda::pipeline<thread_scope_thread>` because each thread owns its
   own ring slot (no cross-thread sharing). For warp-cooperative or
   block-cooperative loads (multiple lanes filling a single staged tile),
   `thread_scope_block` is required and uses `__pipeline_wait_prior`
   under the hood — but at TPB=16 with each lane reading its own
   16 B stripe, there's no cross-thread sync to do.
3. **`#pragma unroll` on the pipeline loop CAN double-unroll the
   `producer_acquire/commit` calls.** We marked the steady-state loop
   `#pragma unroll 4` (not full unroll) to avoid blowing up the kernel
   text — full unroll at D_K=256 produced a 5× larger SASS in early
   experiments. At unroll=4 the cp.async commits stay grouped sensibly.
4. **`prologue` variable warning.** The kernel has a dead local
   `int prologue = ...` that the compiler warns about (#177-D). Harmless
   — it was a clarity comment for the prologue intent that the actual
   prologue loop bounded with `s < N_STAGES` already. Left in for
   readability; could be deleted with no functional change.
5. **`__syncthreads()` between Pass 1 and Pass 2 is technically
   unnecessary.** At TPB=16, each lane writes and reads its own column of
   `smem_S`, no cross-thread sharing. Kept the barrier in for
   future-safety: any move to TPB > VLANES would require it. Cost is
   ~1 BAR.SYNC, ≪ 1% of runtime.
6. **TMA / `UTMALDG` is NOT accessible on sm_120 from CUDA C++ or
   `cuda::pipeline`.** Per Wave 22.8: TMA needs sm_100a (Hopper) or
   newer Blackwell-server (sm_103a). RTX 5090 is sm_120 (Blackwell
   gaming) and the cccl headers correctly fall back to cp.async. Do
   not attempt to enable TMA on this hardware — it will silently
   downgrade or fail PTX validation.
7. **Pass-2 STG.E.128 is NOT pipelined.** `cuda::memcpy_async` only
   supports gmem→smem on sm_8x/9x/12x. The smem→gmem path requires
   `cooperative_groups::memcpy_async` with explicit copy descriptors or
   raw `cp.async.bulk` (TMA family) — out of scope here. State-write
   bandwidth remains a synchronous `STG.E.128` chain. If state writes
   become the bottleneck after this win, the next move is double-
   buffering across timesteps in a multi-step decode harness, not
   single-step async stores.
